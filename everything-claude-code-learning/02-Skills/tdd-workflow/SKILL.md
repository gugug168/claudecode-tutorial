# 测试驱动开发工作流

<!--
【说明】此技能确保所有代码开发遵循 TDD 原则并有全面的测试覆盖。
-->
此技能确保所有代码开发遵循 TDD 原则并有全面的测试覆盖。

<!--
【说明】何时激活:
- 编写新功能或特性
- 修复 bug 或问题
- 重构现有代码
- 添加 API 端点
- 创建新组件
-->
## 何时激活

- 编写新功能或特性
- 修复 bug 或问题
- 重构现有代码
- 添加 API 端点
- 创建新组件

<!--
【说明】核心原则
-->
## 核心原则

<!--
【说明】1. 测试先于代码:始终先写测试,然后实现代码使测试通过。
-->
### 1. 测试先于代码

始终先写测试,然后实现代码使测试通过。

<!--
【说明】2. 覆盖率要求:
- 最低 80% 覆盖率(单元 + 集成 + E2E)
- 覆盖所有边缘情况
- 测试错误场景
- 验证边界条件
-->
### 2. 覆盖率要求
- 最低 80% 覆盖率(单元 + 集成 + E2E)
- 覆盖所有边缘情况
- 测试错误场景
- 验证边界条件

<!--
【说明】3. 测试类型:
- 单元测试:单个函数和工具、组件逻辑、纯函数
- 集成测试:API 端点、数据库操作、服务交互
- E2E 测试:关键用户流程、完整工作流、浏览器自动化
-->
### 3. 测试类型

#### 单元测试
- 单个函数和工具
- 组件逻辑
- 纯函数

#### 集成测试
- API 端点
- 数据库操作
- 服务交互

#### E2E 测试 (Playwright)
- 关键用户流程
- 完整工作流
- 浏览器自动化

<!--
【说明】TDD 工作流步骤
-->
## TDD 工作流步骤

<!--
【说明】步骤1:编写用户旅程
-->
### 步骤 1:编写用户旅程
```
作为 [角色],我想要 [操作],以便 [利益]

示例:
作为用户,我想要语义化搜索市场,
以便即使没有确切关键词也能找到相关市场。
```

<!--
【说明】步骤2:生成测试用例
-->
### 步骤 2:生成测试用例

```typescript
describe('语义搜索', () => {
  it('为查询返回相关市场', async () => {
    // 测试实现
  })

  it('优雅地处理空查询', async () => {
    // 测试实现
  })

  it('当 Redis 不可用时回退到子字符串搜索', async () => {
    // 测试实现
  })
})
```

<!--
【说明】步骤3:运行测试(应该失败)
-->
### 步骤 3:运行测试(应该失败)
```bash
npm test
# 测试应该失败 - 我们尚未实现
```

<!--
【说明】步骤4:实现代码
-->
### 步骤 4:实现代码

```typescript
// 由测试指导的实现
export async function searchMarkets(query: string) {
  // 实现代码
}
```

<!--
【说明】步骤5:再次运行测试
-->
### 步骤 5:再次运行测试
```bash
npm test
# 测试现在应该通过
```

<!--
【说明】步骤6:重构:在保持测试绿色的同时提高代码质量
- 移除重复
- 改进命名
- 优化性能
-->
### 步骤 6:重构

在保持测试绿色的同时提高代码质量:
- 移除重复
- 改进命名
- 优化性能

<!--
【说明】步骤7:验证覆盖率
-->
### 步骤 7:验证覆盖率
```bash
npm run test:coverage
# 验证达到 80%+ 覆盖率
```

<!--
【说明】最佳实践:
1. **先写测试** - 始终 TDD
2. **每个测试一个断言** - 关注单一行为
3. **描述性测试名称** - 解释测试内容
4. **Arrange-Act-Assert** - 清晰的测试结构
5. **Mock 外部依赖** - 隔离单元测试
6. **测试边缘情况** - null、undefined、空、大
7. **测试错误路径** - 不仅是正常路径
-->
## 最佳实践

1. **先写测试** - 始终 TDD
2. **每个测试一个断言** - 关注单一行为
3. **描述性测试名称** - 解释测试内容
4. **Arrange-Act-Assert** - 清晰的测试结构
5. **Mock 外部依赖** - 隔离单元测试
6. **测试边缘情况** - null、undefined、空、大
7. **测试错误路径** - 不仅是正常路径

<!--
【说明】成功指标:
- 达到 80%+ 代码覆盖率
- 所有测试通过(绿色)
- 没有跳过或禁用的测试
- 快速测试执行(单元测试 < 30秒)
- E2E 测试覆盖关键用户流程
-->
## 成功指标

- 达到 80%+ 代码覆盖率
- 所有测试通过(绿色)
- 没有跳过或禁用的测试
- 快速测试执行(单元测试 < 30秒)
- E2E 测试覆盖关键用户流程
